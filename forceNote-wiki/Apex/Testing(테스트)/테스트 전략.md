# 테스트 전략

> Apex 테스트의 구조화 원칙. 단위 테스트 격리, 외부 의존성 모킹, 체계적인 검증.

---

## 개념

### 왜 Apex 테스트 전략이 중요한가

Salesforce는 배포 시 **코드 커버리지 75% 이상**을 강제한다. 그러나 커버리지 달성만을 목표로 한 테스트는 실제 버그를 잡지 못한다. 잘 설계된 테스트 전략은 다음을 보장한다.

- **회귀 방지**: 새 기능 추가나 리팩터링 시 기존 동작이 깨지지 않음을 빠르게 확인
- **배포 안전성**: Sandbox → Production 배포 시 릴리즈 유효성 검증
- **설계 피드백**: 테스트하기 어려운 코드는 대부분 설계 문제(강결합, 정적 의존성)가 있다는 신호

### Apex 테스트의 핵심 제약

Apex 테스트 환경에는 일반 코드와 다른 제약이 있다.

- **외부 Callout 불가**: 테스트에서 실제 HTTP 요청을 보내면 예외 발생 → `HttpCalloutMock` 필수
- **SOSL 결과 없음**: Full-Text Search 인덱스가 갱신되지 않으므로 항상 빈 결과 → `Test.setFixedSearchResults()` 필수
- **Governor Limit 별도 집계**: 테스트 데이터 생성(`@TestSetup`, `insert`) 과정이 Limit을 소모하므로, 실제 테스트 대상 코드의 Limit과 분리하기 위해 `Test.startTest()` / `Test.stopTest()`로 경계를 명확히 해야 한다
- **비동기 실행 동기화**: `@future`, `Queueable`, `Batch` 작업은 `Test.stopTest()` 호출 시점에 동기적으로 실행된다

### 테스트 격리 원칙

좋은 Apex 테스트는 **외부 상태에 의존하지 않는다.** 테스트 데이터는 테스트 내부에서 생성하고, 외부 시스템(HTTP, 다른 Apex 클래스의 부수 효과)은 모킹한다. 하나의 테스트 메서드가 다른 테스트의 성공에 영향을 주어서는 안 된다.

## 테스트 클래스 기본 구조

```apex
@isTest
private class BookApiServiceTest {

    // 공통 데이터 초기화
    @TestSetup
    static void makeData() {
        Account acct = new Account(Name = 'Test Account');
        insert acct;
    }

    // 정상 케이스
    @isTest
    static void searchBooks_returnsResults_withValidKeyword() {
        Test.setMock(HttpCalloutMock.class, new SuccessCalloutMock());

        Test.startTest();
        List<BookModel> result = new BookApiService().searchBooks('salesforce');
        Test.stopTest();

        Assert.isFalse(result.isEmpty(), '결과가 있어야 함');
        Assert.areEqual('Salesforce Book', result[0].title);
    }

    // 실패 케이스
    @isTest
    static void searchBooks_throwsException_on404() {
        Test.setMock(HttpCalloutMock.class, new ErrorCalloutMock(404, '{}'));

        Test.startTest();
        try {
            new BookApiService().searchBooks('unknown');
            Assert.fail('예외가 발생해야 함');
        } catch (BookApiService.BookNotFoundException e) {
            Assert.isNotNull(e.getMessage());
        }
        Test.stopTest();
    }
}
```

## 의존성별 모킹 전략

| 의존성 유형 | 모킹 방법 |
|---|---|
| HTTP 외부 호출 | `HttpCalloutMock` + `Test.setMock()` |
| Apex 클래스 메서드 | `StubProvider` + `Test.createStub()` |
| static 메서드 | `@testVisible` Boolean/Exception 회로차단기 |
| Salesforce 시스템 클래스 | 래퍼 클래스 → StubProvider |
| ConnectApi | 래퍼 클래스 → StubProvider |
| Quiddity | `QuiddityGuard.testQuiddityOverride` |

## Assert 모범 사례

```apex
// ✅ 명시적 메시지 포함
Assert.areEqual(expected, actual, '설명 메시지');
Assert.isTrue(condition, '이 조건이 true여야 하는 이유');

// ✅ 예외 테스트는 fail() 포함
try {
    methodThatShouldThrow();
    Assert.fail('예외가 발생해야 함'); // 예외 없으면 테스트 실패
} catch (ExpectedException e) {
    Assert.isNotNull(e.getMessage());
}

// ❌ 피해야 할 패턴
System.assertEquals(expected, actual); // 구형 API
Assert.isTrue(list.size() > 0);        // isEmpty() 사용 권장
Assert.areEqual(1, list.size());       // 더 명확
```

## Test.startTest / Test.stopTest 사용

```apex
// Governor limit 초기화 — 외부 callout 포함 시 필수
Test.startTest();
    List<BookModel> result = service.searchBooks('sf'); // callout 실행
Test.stopTest();
// stopTest()에서 @future, Queueable, Batch가 동기 실행됨
```

> [!tip] startTest/stopTest 역할
> - Governor limit을 초기화하여 테스트 자체의 DML/쿼리를 테스트 대상에서 분리
> - `stopTest()` 호출 시 비동기 작업(@future, Queueable, Batch)이 강제 실행됨

## @TestSetup vs 각 테스트 내 데이터 생성

| | @TestSetup | 각 테스트 내 생성 |
|---|---|---|
| 실행 시점 | 테스트 클래스 실행 전 1회 | 각 테스트 메서드마다 |
| 격리 | 각 테스트는 독립된 사본 | 완전 격리 |
| 성능 | 빠름 (공유 데이터) | 느림 |
| 권장 | 공통 기준 데이터 | 테스트 고유 데이터 |
